# oxr-sdk는 AI Ready인가?

> **결론: Yes. 프롬프트만으로 작동하는 협업 룸이 나온다.**

---

## 발표 흐름 요약 (3줄)

1. **왜 중요한가**: AI Ready 프레임워크는 업계 용어·내부 규칙을 AI가 대신 알게 한다 — 숙련 개발자가 아닌 **누구나** 자연어만으로 XR 룸을 만들 수 있는 세상으로의 전환점이다.
2. **전환 + 증거**: XRCollabDemo는 이제 "받아서 쓰는 샘플"이 아니라 프롬프트로 조립되는 결과물이다. 사람의 암묵지 없이 AI가 작동 룸을 빌드·검증했다 (`Client 0 is successfully validated`, `NetworkObjects=5`).
3. **제안**: oxr-sdk는 AI Ready 기준을 충족한다 — XRCollabDemo를 best/reference sample로 채택하고, 샘플 제공 방식을 정적 배포에서 프롬프트형 제공으로 전환할 수 있다.

---

## 1. 'AI Ready'란?

**기준 정의:** AI 에이전트가 문서와 계약만으로, 이 프레임워크 위에서 작동하는 콘텐츠를 조립하고 스스로 검증할 수 있는가.

- 단순히 "AI가 코드를 짤 수 있다"는 수준이 아니다.
- 프레임워크의 규칙이 **명시적**이고, 확장 면이 **표준화**되어 있고, 검증 기준이 **구조화**되어 있어야 한다.
- 그래야만 AI가 사람의 암묵지 없이도 작동하는 결과물을 재현하고, 스스로 맞고 틀림을 판단할 수 있다.

---

## 2. 왜 AI Ready가 중요한가

### 몰라도 만들 수 있다 — 하지만 오류에서 헤맨다

`NetworkObject`, `XumPlayerSpawner`, `_offlineScene`, C1~C4 불변식 — 이것들을 **반드시 알아야만 룸을 만들 수 있는 건 아니다.** 하지만 모르면 이런 일이 벌어진다:

- 스포너를 스크립트로 만든다 → `"Failed to confirm the access"` → 왜인지 모름 → 몇 시간 디버깅
- `_offlineScene`을 빈칸으로 둔다 → 룸에 입장해도 로비 UI가 덮음 → 어떤 설정이 문제인지 모름
- 컬렉션을 다른 걸로 설정한다 → 입장은 되는데 아바타가 안 보임 → 원인 추적 불가

**알아야 만들 수 있는 게 아니라, 모르면 오류 → 시행착오 → 시간 낭비의 반복이다.**  
AI Ready 프레임워크는 이 함정들을 AI가 대신 알고 있어서, 첫 시도부터 작동하는 결과가 나온다.

### 그럼 샘플 클론(clone)이랑은 뭐가 다른가

"XRCollabDemo를 clone해서 쓰면 되는 거 아닌가?"라는 질문이 나올 수 있다. 차이는 이렇다:

| | 샘플 클론 방식 | AI Ready 방식 |
|---|---|---|
| **시작점** | 누군가 만들어 놓은 결과물을 복사 | 프롬프트 → AI가 처음부터 조립 |
| **커스터마이징** | 내부 구조를 이해해야만 수정 가능 | 자연어로 원하는 것 설명 → AI가 반영 |
| **오류 발생 시** | 원인을 스스로 찾아야 함 | AI가 불변식(C1~C4) 기준으로 진단 |
| **새 기능 추가** | 코드 구조를 파악한 뒤 직접 작성 | 계약(`IToggleableContent`)에 맞게 AI가 생성 |
| **한계** | 샘플의 범위 안에서만 응용 가능 | 새로운 룸을 설명만으로 조립 가능 |

클론은 **출발점을 빠르게 주지만**, 거기서 한 발만 벗어나면 다시 내부 지식이 필요해진다.  
AI Ready는 **내부 지식 자체를 AI가 갖고 있어서**, 사용자는 원하는 결과를 말하는 것으로 충분하다.

### 결과: 도달 가능한 사용자의 범위가 바뀐다

- **이전:** Unity + oxr-sdk를 깊이 아는 숙련 개발자만 새 룸을 만들 수 있다.
- **이후:** 자연어로 소통할 수 있는 **누구나** — 연구자, 기획자, 비개발자도 — 룸을 만들 수 있다.

> *"여러 사람이 들어와서 같이 거리를 잴 수 있는 방 만들어줘"*  
> → AI가 `IToggleableContent`, FishNet 씬 규칙, C1~C4를 알아서 처리한다. 사용자는 업계 용어를 배울 필요가 없다.

## 3. 패러다임 전환 — Before → After

| | Before | After |
|---|---|---|
| **XRCollabDemo 역할** | 받아서 쓰는 **정적 샘플 프로젝트** | oxr-sdk 기반 **프롬프트형 생성** 결과물 |
| **사용 방식** | clone → 그대로 실행 | 프롬프트 → AI가 조립 → 동일 결과물 |
| **암묵지 의존** | 개발자가 직접 알아야 함 | 문서·계약으로 대체 가능 |
| **확장** | 코드 직접 수정 | 계약 위에 기능 모듈 추가 |

핵심 전환: "우리가 만들어 놓은 샘플"에서 **"우리 프레임워크라면 AI도 이걸 만든다"** 로.

## 4. ⭐ 증거 — 문서/프롬프트만으로 작동 룸 재현 (BasicRoom 실측)

### 실행 환경

| 구성 요소 | 내용 |
|---|---|
| **AI 에이전트** | Claude Code (claude-sonnet-4-6) |
| **Unity 연동** | MCP(Model Context Protocol) — Unity Editor에 포트 27826으로 연결 |
| **입력** | `build-working-room.md` 문서 + 자연어 프롬프트 |
| **Unity 프로젝트** | XRCollabDemo (Unity 6000.3.11f1) |

Claude Code가 MCP를 통해 Unity Editor를 직접 조작 — 씬 생성, 프리팹 배치, 빌드, 로그 확인까지 전부 AI가 수행.

### "사람의 암묵지 없이, 문서만 보고 AI가 빌드 → 실행 → 검증"

`build-working-room.md` **하나만 보고** BasicRoom을 처음부터 조립 → 빌드 → 실행 → 검증.

**실측 결과 (AI가 만든 결과):**

| 검증 항목 | 결과 |
|---|---|
| 서버 룸 등록 | `Online Scene: BasicRoom` + **`Client 0 is successfully validated`** |
| 로비 자동 소멸 | `Client` 씬 언로드 / `C-MasterCanvas` 소멸 (UI 자동 전환) |
| 네트워크 오브젝트 | **`NetworkObjects=5`**, `Desktop(Clone)` + `mixamorig:Hips` 모두 `scene=BasicRoom` |
| 아바타 스폰 | 카메라 활성, 게임뷰에 룸 표시, 로비 가림 없음 |

**의미:** 패키지 README나 플랫폼 공식 문서만으로는 재현이 안 됐다. AI가 문서의 빈틈(스포너 구성, 불변식)을 발굴해 채운 덕분에, 비로소 처음 보는 사람도 재현할 수 있는 문서가 완성됐다.

## 5. 결론 & 제안

### oxr-sdk는 AI Ready 기준을 충족한다

- 작동 규칙이 **명시적**으로 문서화됐다 (C1~C4).
- 확장 면이 **표준 계약**으로 정의됐다 (`IToggleableContent`).
- 검증 기준이 **구조화**됐다 (하네스 + 실측 수치).
- 사람의 암묵지 없이 AI가 처음부터 작동 룸을 재현했다 (BasicRoom 실측).

### 제안 1: XRCollabDemo를 best/reference sample로 채택

- 더 이상 "다운로드하는 정적 샘플"이 아니다.
- oxr-sdk 위에서 프롬프트로 조립되는 결과물 — 즉 **프레임워크의 AI 준비도를 증명하는 레퍼런스**다.
- 우리 프레임워크 기반이라면 이 정도 결과가 나온다는 것을 공식 인정.

### 제안 2: 샘플 제공 방식 전환

| 현재 | 제안 |
|---|---|
| XRCollabDemo 저장소 clone + 설정 | oxr-sdk + 프롬프트 → 동일 결과 조립 |
| 정적 배포 | **프롬프트형 제공** |

## 6. 로드맵 — AI 저작 파이프라인 완성으로 가는 길

| Phase | 내용 | 상태 |
|---|---|---|
| 0 | 룸 베이스 (네트워크+아바타+UI전환, C1~C4) | ✅ 완료 |
| 1 | 계약 규격 (`IToggleableContent`, 씬 계층 표준) | ✅ 완료 |
| 2 | Ruler 파일럿 (계약 위에 클린 재구현·검증) | ✅ 완료 |
| 2.5 | 씬 계층 표준화 (SYSTEMS/ENVIRONMENT/UI/FEATURES/_DYNAMIC) | ✅ 완료 |
| **3** | **런치패드 UI** — registry → 아이콘 그리드 → `SetEnabled` | 예정 |
| **4** | **스킬화** — `/RoomContent-<feature>` + LLM 신규기능 생성 템플릿 | 예정 |
| **5** | **합성+하네스** — `/promptscene`(자연어→룸 조립) + C1~C4/콘텐츠 자동 검증 | 예정 |

Phase 3~5가 완성되면: 자연어 프롬프트 한 줄 → 작동하는 협업 룸 + 자동 검증 리포트.

## 7. 참고

### 재현 환경
- **AI:** Claude Code (claude-sonnet-4-6) + MCP → Unity Editor (포트 27826)
- **프로젝트:** XRCollabDemo, Unity 6000.3.11f1, com.oxr.sdk embed
- **서버:** Master `192.168.50.49:5000`, Room `:7777`

### 핵심 용어

| 용어 | 설명 |
|---|---|
| **NetworkObject** | FishNet 네트워크 동기화 단위. 씬에 배치된 것은 프리팹 인스턴스화로만 유효한 id를 받는다. |
| **스포너** | 플레이어 입장 시 아바타를 소환하는 씬 오브젝트 (`XumPlayerSpawner`). 반드시 프리팹으로 인스턴스화해야 한다. |
| **IToggleableContent** | oxr-sdk 콘텐츠 플러그인 계약. 이 인터페이스 하나로 자기등록·토글·메타 제공이 가능하다. 코어는 이 기능의 존재를 모른다. |
