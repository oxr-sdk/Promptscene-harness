# OXR-SDK는 AI-Ready 되었는가

> 발표 본문 초안. 목적: XRCollabDemo 성과 보고가 아니라, OXR-SDK 계열 프레임워크가 AI-ready한 방식으로 제공될 수 있음을 주장하고 검증한다.

---

## 1) 주장

오늘 발표의 질문은 단순합니다.

> **Is OXR-SDK AI-ready?**

이 발표의 주장은 다음과 같습니다.

> **OXR-SDK는 AI-ready한 프레임워크로 전환될 수 있는 구조를 갖고 있었고, 이번 작업을 통해 그 구조가 문서·프롬프트·검증 기준으로 실제 작동함을 확인했습니다.**

이번 검증은 XRCollabDemo 위에서 수행했습니다. 이유는 XRCollabDemo가 OXR-SDK의 여러 레이어가 실제 서비스 흐름으로 맞물리는 대표 샘플이기 때문입니다. 로비, 룸 생성, 네트워크 스폰, 아바타, 씬 전환이 모두 포함되어 있어서, OXR-SDK가 AI-ready한 방식으로 제공될 수 있는지를 확인하기에 좋은 검증 무대였습니다.

따라서 이번 발표는 XRCollabDemo 자체의 완성도를 보고하는 발표라기보다, 다음 질문에 답하는 발표입니다.

> **OXR-SDK 프레임워크 위에서 XRCollabDemo 수준의 협업 룸이 정적 샘플 클론 없이 재생성될 수 있는가?**

조금 더 엄밀하게 말하면, 기존 문서가 처음부터 완벽했다는 뜻은 아닙니다. 기존 문서와 패키지는 AI가 사용할 수 있는 충분한 구조적 재료를 제공하고 있었지만, AI 에이전트가 그대로 실행하기에는 일부 암묵지가 남아 있었습니다. 이번 작업은 그 암묵지를 절차와 검증 기준으로 보강하고, 실제로 AI가 새 룸을 만들고 빌드하고 검증할 수 있음을 확인한 과정입니다.

---

## 2) AI-Ready의 정의

여기서 말하는 AI-ready는 "AI가 마법처럼 모든 것을 알아서 만든다"는 뜻이 아닙니다.

이 발표에서는 AI-ready를 다음과 같이 정의합니다.

> **SDK가 가진 기능과 사용 절차가 문서, 패키지, 프롬프트, 검증 기준으로 정리되어 있어서, AI 에이전트가 사람의 암묵지에 의존하지 않고 작동하는 산출물을 만들고 검증할 수 있는 상태.**

따라서 AI-ready는 단순히 문서가 많다는 뜻이 아닙니다. AI-ready는 다음 조건을 요구합니다.

- AI가 읽고 실행할 수 있는 절차가 있어야 합니다.
- 반드시 지켜야 하는 작동 조건이 명시되어야 합니다.
- 결과가 맞는지 확인할 수 있는 검증 기준이 있어야 합니다.
- 샘플은 실행 결과물이 아니라 재생성 가능한 레퍼런스가 되어야 합니다.

이번 발표에서 증명하는 것은 "보강된 절차 문서와 패키지만으로 작동 룸을 자율로 빌드·검증했다"는 것입니다. "기존 문서가 처음부터 완전했다"거나 "문장 하나로 임의의 룸을 합성한다"는 주장은 아닙니다.

---

## 3) 한 장 요약 — AI-Ready 전과 후

이번 발표에서 가장 보여줘야 하는 차이는 이것입니다.

| 구분 | 기존 샘플 방식 | AI-ready 방식 |
|---|---|---|
| 제공 형태 | XRCollabDemo 프로젝트 다운로드 | OXR-SDK 프레임워크 + 문서 + 프롬프트 + 검증 기준 |
| 지식 위치 | 샘플 씬과 프리팹 안에 암묵적으로 포함 | 절차, 불변식, 프리팹 규칙, 씬 토폴로지로 외부화 |
| AI 역할 | 기존 프로젝트를 열어 보고 따라 함 | 새 룸을 조립하고 빌드하고 로그로 검증 |
| 실패 대응 | 왜 실패했는지 사람이 디버깅 | 어떤 조건이 깨졌는지 기준으로 판정 |
| 재사용성 | 샘플 밖으로 나가면 다시 분석 필요 | 다른 룸과 기능으로 확장 가능 |
| 의미 | "샘플이 작동한다" | "프레임워크가 AI에게 작동한다" |

핵심은 샘플을 없애는 것이 아닙니다. 오히려 XRCollabDemo는 계속 중요합니다. 다만 그 위상이 바뀝니다. 기존에는 받아서 실행하는 정적 샘플이었다면, 이제는 OXR-SDK의 AI-ready 가능성을 보여주는 **재생성 가능한 best sample**로 사용할 수 있습니다.

## 4) OXR-SDK 구성요소

OXR-SDK는 하나의 단일 기능이 아니라 여러 프레임워크 레이어가 맞물리는 구조입니다.

| 구성요소 | 역할 |
|---|---|
| `com.oxr.sdk` | OXR Portal Backend REST API 래퍼. 인증, 에셋 스토어, 파일 스토리지, 룸 관리 제공 |
| `XumLobby` | Master Server Toolkit 기반 인증, 로비 UI, 룸 생성, 룸 진입, 룸 씬 전환 |
| `XumNet` | 네트워크 오브젝트 생성과 스폰을 감싸는 레이어. `XumNetwork.Instantiate` 기반 스폰 |
| `UnifiedXRMotion` | 플랫폼별 아바타와 모션. Desktop/UnityXR 아바타 스폰과 이동 검증의 기준 |

따라서 AI-ready 여부도 "데모 프로젝트 하나가 실행되는가"가 아니라, 이 레이어들이 AI가 읽고 조립할 수 있는 형태로 제공되는가로 봐야 합니다.

정적 샘플 방식에서는 이 레이어들이 이미 맞물린 결과물로 제공됩니다. XRCollabDemo를 다운로드하면 빠르게 실행할 수 있지만, 작동 지식은 프로젝트 내부에 숨어 있습니다. 반면 AI-ready 방식에서는 이 작동 지식이 문서, 계약, 프롬프트, 검증 기준으로 밖에 나와 있어야 합니다. 그래야 AI가 기존 샘플을 복사하는 것이 아니라, 프레임워크를 사용해 새 산출물을 재생성할 수 있습니다.

## 5) 검증 방법

검증은 세 단계로 진행했습니다.

첫째, OXR-SDK 패키지와 관련 문서를 읽고, 실제로 작동하는 룸을 만들기 위해 필요한 조건을 하나의 절차 문서로 정리했습니다. 기존 문서와 README는 OXR-SDK의 주요 기능과 샘플 흐름을 제공하고 있었지만, AI 에이전트가 처음부터 룸을 조립하기에는 몇 가지 조건이 더 명시되어야 했습니다.

둘째, 작동에 필요한 조건을 C1~C4 불변식으로 정리했습니다.

| 조건 | 내용 |
|---|---|
| C1 | 서버와 클라이언트의 네트워크 프리팹 컬렉션이 일치해야 함 |
| C2 | 플레이어 스폰은 지정된 스포너 프리팹과 플랫폼별 아바타 카탈로그를 통해 이뤄져야 함 |
| C3 | 룸 진입과 이탈 시 로비 씬과 룸 씬이 정확히 전환되어야 함 |
| C4 | 서버는 Master+Room 실행파일, 에디터는 Client+Room 흐름을 기준으로 검증해야 함 |

셋째, AI 에이전트와 MCP만으로 이 절차를 수행했습니다. AI가 Unity 에디터를 직접 조작해 씬을 만들고, 네트워크 프리팹을 배치하고, 스포너 프리팹을 인스턴스화하고, 서버 실행파일을 빌드하고, 서버 로그와 런타임 상태를 확인했습니다.

이 작업은 OXR-SDK 자체의 기능을 새로 만든 것이 아닙니다. 이미 존재하던 프레임워크의 사용 조건을 AI가 실행할 수 있는 형태로 정리한 것입니다. 즉, **documentation hardening**이자 **AX 관점의 절차화**입니다.

검증 환경은 다음과 같습니다.

| 항목 | 값 |
|---|---|
| AI 에이전트 | Claude Code (claude-sonnet-4-6) |
| Unity 연동 | MCP → Unity Editor, 포트 27826 |
| 대상 프로젝트 | XRCollabDemo |
| Unity 버전 | 6000.3.11f1 |

## 6) 검증 결과

검증 결과는 "그럴듯한 룸이 보였다"가 아니라, 실제 서버 로그와 Unity 런타임 상태에서 확인했습니다.

첫째, 룸이 서버에 정상 등록되었습니다. 서버 로그에 `Online Scene: BasicRoom`이 출력되었고, 클라이언트는 `Client 0 is successfully validated`로 인증되었습니다. 마스터 서버는 `192.168.50.49:5000`에서 listening 상태였고, 룸 서버는 `:7777`에서 방을 등록했습니다. 실제 등록된 방은 `Room-1A5C-ED20-3935`, ID 0, 최대 10명, Public이었습니다.

둘째, 로비에서 룸으로 씬 전환이 정상 동작했습니다. 클라이언트의 `Client` 씬이 언로드되고 `BasicRoom`이 로드되었으며, 로비 UI인 `C-MasterCanvas`가 사라졌습니다. 별도 수동 처리 없이 룸 입장 흐름이 이어졌습니다.

셋째, 아바타가 룸에 정상 스폰되었습니다. 네트워크 오브젝트는 5개(`NetworkObjects=5`)였고, 소유 아바타 `Desktop(Clone)`, 골격 `mixamorig:Hips`, 플랫폼 루트가 모두 `scene=BasicRoom`에 위치했습니다. 아바타 카메라가 활성화되어 게임뷰에 룸이 표시되었고, 로비 화면이 남아 룸을 가리는 문제도 없었습니다.

이 결과가 중요한 이유는 단순히 BasicRoom 하나가 만들어졌기 때문이 아닙니다. AI가 OXR-SDK의 로비, 룸 전환, 네트워크 스폰, 아바타 레이어를 따라 실제 작동 결과를 만들고, 그 결과를 로그와 런타임 상태로 검증했기 때문입니다.

별도로, 향후 콘텐츠 확장을 위한 파일럿으로 Ruler 기능도 구현했습니다. `IToggleableContent` 계약 위에서 측정 기능을 등록하고 활성화했으며, 거리 라벨(`4.00m`) 렌더를 확인했습니다. 다만 이것은 OXR-SDK의 AI-ready 판정 근거라기보다, 이후 자연어 기반 콘텐츠 확장을 위한 설계 실험으로 보는 것이 맞습니다.

## 7) 증명 매핑 — 기준과 증거

위 결과가 AI-ready 주장과 연결되는 지점은 다음과 같습니다.

| AI-ready 판단 기준 | 실제 검증 증거 | 의미 |
|---|---|---|
| AI가 절차를 따라 룸을 만들 수 있는가 | MCP로 `BasicRoom` 생성, 네트워크 프리팹 배치, 서버 빌드 수행 | 문서와 절차가 실행 가능한 형태로 정리됨 |
| 서버에 룸이 등록되는가 | `Online Scene: BasicRoom`, `Room registered successfully` | XumLobby의 룸 서버/등록 흐름이 작동함 |
| 클라이언트 입장이 검증되는가 | `Client 0 is successfully validated` | 인증과 룸 입장 흐름이 연결됨 |
| 로비에서 룸으로 전환되는가 | `Client` 언로드, `BasicRoom` 로드, `C-MasterCanvas` 제거 | 로비 UI와 룸 씬 전환 조건이 맞음 |
| 아바타가 룸에 스폰되는가 | `NetworkObjects=5`, `Desktop(Clone)`과 `mixamorig:Hips`가 `scene=BasicRoom`에 위치 | XumNet 스폰과 UnifiedXRMotion 아바타 레이어가 연결됨 |
| AI가 결과를 판정할 수 있는가 | C1~C4 기준으로 서버 로그와 런타임 상태 확인 | 단순 실행이 아니라 검증 가능한 절차가 됨 |

이 표가 결론의 핵심입니다. AI-ready 전에는 "샘플이 작동한다"는 사실만 있었습니다. AI-ready 방식에서는 "어떤 조건을 만족했기 때문에 작동한다고 판단하는지"까지 드러납니다.

## 8) 결론

결론을 한 문장으로 말하면 다음과 같습니다.

> **OXR-SDK는 AI-ready한 프레임워크로 전환될 수 있는 구조를 갖고 있었고, 이번 작업을 통해 룸 생성·입장·아바타 스폰 범위에서 그 가능성이 실제로 검증되었습니다.**

이 결론은 두 가지를 동시에 말합니다.

첫째, OXR-SDK 계열 프레임워크는 AI-ready가 될 수 있는 충분한 구조를 갖고 있습니다. 포털 API, 로비/룸 전환, 네트워크 스폰, 플랫폼별 아바타가 분리된 레이어로 존재하고, 이 레이어들을 조합하면 실제 협업 룸 흐름을 만들 수 있습니다.

둘째, AI-ready는 단순히 패키지가 존재한다고 자동으로 완성되는 것이 아닙니다. AI가 읽을 수 있는 절차, 작동 불변식, 검증 기준이 함께 있어야 합니다. 이번 작업은 그 지점을 보강했고, 보강된 문서와 프롬프트 기반 절차가 실제 AI 에이전트에게 동작한다는 것을 확인했습니다.

따라서 발표의 메시지는 "XRCollabDemo를 AI로 잘 만들었다"가 아닙니다.

> **OXR-SDK 프레임워크가 AI-ready한 방식으로 제공될 수 있고, XRCollabDemo는 그 가능성을 보여주는 best sample이다.**

앞으로 XRCollabDemo는 정적 다운로드 샘플로만 볼 필요가 없습니다. OXR-SDK의 문서, 프롬프트, 검증 기준과 함께 제공될 때, XRCollabDemo는 재생성 가능한 레퍼런스가 됩니다. 이것이 AI-ready 전과 후의 가장 큰 차이입니다.

## 9) 향후 계획

다음 단계는 룸을 만드는 수준을 넘어, 자연어로 기능을 붙이는 단계입니다.

DeepChairProject처럼 "측정 기능을 넣어줘", "의자를 선택하고 배치하게 해줘" 같은 요구를 자연어로 설명하면, AI가 OXR-SDK 위에 기능을 생성하고 검증하는 흐름을 목표로 합니다.

이를 위해서는 두 가지가 더 필요합니다.

- 콘텐츠 기능을 자유롭게 붙일 수 있는 확장 계약
- 생성된 기능을 사용자가 켜고 끌 수 있는 제너레이티브 UI 구조

현재 Ruler와 `IToggleableContent` 파일럿은 이 다음 단계를 위한 실험입니다. 지금의 핵심 검증은 룸 생성·입장·아바타 스폰이지만, 앞으로는 자연어 기반 콘텐츠 생성과 UI 확장까지 이어가야 합니다.

로드맵은 다음과 같습니다.

| Phase | 내용 | 상태 |
|---|---|---|
| 0 | 룸 베이스 검증: 네트워크, 아바타, UI 전환 | 완료 |
| 1 | 룸 조립 계약 정리: C1~C4, 스포너 규칙, 씬 토폴로지 | 완료 |
| 2 | AI + MCP 기반 룸 생성·빌드·검증 | 완료 |
| 3 | 콘텐츠 확장 파일럿: `IToggleableContent`, Ruler | 완료 |
